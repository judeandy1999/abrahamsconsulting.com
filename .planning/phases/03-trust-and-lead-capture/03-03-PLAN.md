<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public form input->`POST /api/lead` | Untrusted user input crosses into server-side processing and outbound email logic. |
| server route->email provider API | Trusted backend sends lead details to third-party delivery infrastructure. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Spoofing | lead submissions | mitigate | Validate payload schema, require essential qualification fields, and reject honeypot-triggered submissions. |
| T-03-08 | Tampering | API payload/body | mitigate | Enforce strict server-side parsing and never trust client-only validation. |
| T-03-09 | Information Disclosure | email credentials and lead data | mitigate | Keep provider keys in server env vars only and avoid exposing secrets in client bundles or logs. |
| T-03-10 | Denial of Service | lead endpoint abuse | mitigate | Add anti-spam controls plus bounded error handling; return deterministic rejection for malformed/abusive requests. |
| T-03-SC | Tampering | package/dependency supply chain | mitigate | Prefer direct HTTP integration to avoid new packages; if an SDK is added, run package legitimacy checkpoint first. |
</threat_model>
